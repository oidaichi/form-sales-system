# フォーム営業AIエージェント要件定義書

## 1. システム概要

### 1.1 目的
エクセルファイルに記載された企業ホームページURLから自動的に問合せフォームを検出し、営業メールを自動送信するAIエージェントシステムを構築する。

### 1.2 システム名
フォーム営業AIエージェント

### 1.3 対象企業
日本企業のみ

### 1.4 処理規模
1回の実行で100社～1000社のホームページを処理

## 2. 機能要件

### 2.1 コア機能

#### 2.1.1 エクセルファイル処理機能
- **ファイルアップロード**: 手動アップロード機能
- **列構成管理**: ユーザーが都度列構成を指定可能
- **設定記憶**: 過去の列構成設定を記憶・再利用
- **前回ファイル参照**: 一度指定したエクセルファイルの自動参照
- **重複処理**: エクセル行の重複チェック・除外機能

#### 2.1.2 Webクローラー機能
- **ホームページ巡回**: 企業ホームページ内の問合せフォーム検索
- **フォーム検出**: 一般的な問合せフォームの自動検出
- **代替手段**: フォーム未検出時のメールアドレス抽出
- **JavaScript対応**: 数秒待機後のJavaScript動的フォーム検出
- **CAPTCHA対応**: reCAPTCHA等の認証システム対応（必須）

- **配信停止**: 企業からの停止要請に基づく配信停止機能
- **停止リスト管理**: 配信停止企業の管理機能

### 2.2 外部連携機能

#### 2.2.1 CRM/MAツール連携
- **対象ツール**:
  - Salesforce
  - HubSpot
  - Marketo
  - Google スプレッドシート
- **API連携**: 各ツールとのAPI連携機能
- **データ形式設定**: 連携時のデータ形式ユーザー設定
- **送信タイミング設定**: 連携実行タイミングの設定機能

#### 2.2.2 ChatGPT API連携
- **メール生成**: 企業別営業メール本文の自動生成
- **HTMLメール作成**: HTMLメールテンプレートの動的生成

### 2.3 管理・監視機能

#### 2.3.1 ログ管理機能
- **送信結果記録**: 全送信結果の詳細ログ
- **エラーログ**: フォーム未検出、送信失敗等のエラー記録
- **エクセル出力**: ログ情報のエクセル列への自動記載
- **ログ表示**: ツール内でのログ確認機能

#### 2.3.2 進捗管理機能
- **リアルタイム表示**: 処理中の進捗状況表示
- **処理状況通知**: 処理完了・エラー発生時のメール通知

#### 2.3.3 データエクスポート機能
- **CSV出力**: 必要なデータのCSVダウンロード機能
- **結果出力**: 送信結果・ログデータの出力機能

### 2.4 分析機能

#### 2.4.1 開封率分析
- **Google Analytics連携**: トラッキングピクセルによる開封率測定
- **ダッシュボード**: ツール内での分析結果表示

## 3. 非機能要件

### 3.1 実行環境
- **OS**: Ubuntu（ローカル環境）
- **データベース**: ファイルベース（SQLite等）またはデータベースレス構成
- **認証**: 認証機能不要（単一ユーザー使用想定）

### 3.2 性能要件
- **処理能力**: 100～1000社/回の処理対応
- **送信制限**: ユーザー指定可能な1日当たり送信件数制限
- **応答時間**: フォーム検出時のJavaScript待機時間（数秒）

### 3.3 セキュリティ要件
- **個人情報保護**: 企業情報の適切な管理
- **配信停止対応**: 配信停止要請への迅速な対応
- **重複防止**: 同一企業への重複送信防止

### 3.4 可用性要件
- **エラーハンドリング**: 各種エラー状況への適切な対応
- **ログ保存**: 全処理ログの永続化
- **復旧機能**: 処理中断時の再開機能

## 4. システム構成

### 4.1 主要コンポーネント
1. **エクセル処理モジュール**
2. **Webクローラーモジュール**
3. **フォーム検出・入力モジュール**
4. **メール送信モジュール**
5. **外部API連携モジュール**
6. **ログ管理モジュール**
7. **UI/管理画面モジュール**

### 4.2 データフロー
```
エクセルファイル → 企業URL抽出 → Webクローラー → フォーム検出 
→ 自動入力 → HTMLメール送信 → ログ記録 → CRM連携 → 結果出力
```

## 5. 制約事項

### 5.1 技術的制約
- ローカルUbuntu環境での実行
- CAPTCHA対応の技術的難易度
- JavaScript動的フォームの完全対応困難性
- メール開封率測定の精度限界（画像ブロック影響）

### 5.2 運用制約
- 単一ユーザーでの使用想定
- 手動でのファイルアップロード・設定
- 配信停止要請への手動対応必要

### 5.3 法的制約
- 特定電子メール法の遵守
- 個人情報保護法の遵守
- 配信停止要請への適切な対応

## 6. 今後の拡張可能性

### 6.1 機能拡張
- 多言語対応（海外企業対応）
- AI精度向上によるフォーム検出率向上
- より多くのCRM/MAツールへの対応

### 6.2 技術拡張
- クラウド環境への移行対応
- 複数ユーザー対応
- API化による他システムとの連携

## 7. 成功指標

### 7.1 定量指標
- フォーム検出成功率: 80%以上
- メール送信成功率: 95%以上
- システム処理時間: 1社あたり平均30秒以内
- 上記を確かめるために、コードを実行してテストすること

### 7.2 定性指標
- ユーザーの操作性向上
- 営業効率の向上
- 手動作業時間の削減
